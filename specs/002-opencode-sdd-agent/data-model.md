# Data Model: Unified SDD Agent for OpenCode

## RepositorySetup

- **Purpose**: Represents the managed workflow footprint installed in a repository.
- **Key fields**:
  - `repoRoot`: canonical repository location
  - `entrypointCommand`: primary OpenCode command exposed to users
  - `managedAssets`: set of workflow assets under `.opencode` and `.specify`
  - `mergePolicy`: repository init behavior for pre-existing managed files
  - `agentContextFile`: repo-local guidance file for future agent sessions
  - `status`: current setup state
- **Validation rules**:
  - Must expose exactly one primary guided entrypoint
  - Must include the managed planning asset set required by the workflow
  - Must preserve compatible existing customizations during init
- **State transitions**:
  - `not-initialized` -> `partially-initialized` -> `initialized` -> `upgraded`

## FeatureWorkspace

- **Purpose**: Represents one feature's planning workspace under `specs/`.
- **Key fields**:
  - `workspaceName`: canonical feature workspace identifier
  - `branchName`: suggested or active branch name for the feature
  - `branchType`: change-type prefix such as `feat`, `fix`, `refactor`, `init`, or `test`
  - `shortName`: English kebab-case short name
  - `artifactPaths`: links to `spec.md`, `plan.md`, `tasks.md`, and supporting artifacts
  - `status`: current workflow phase for the feature
- **Validation rules**:
  - `branchType` must come from the allowed prefix set
  - `shortName` must remain filesystem-safe and human-readable
  - Each workspace must own one canonical artifact path per artifact type
- **State transitions**:
  - `discovered` -> `specified` -> `clarified` -> `planned` -> `tasked` -> `archived`

## SDDSession

- **Purpose**: Captures one guided planning interaction across phases.
- **Key fields**:
  - `sessionID`: unique session identifier
  - `activeWorkspace`: feature workspace being planned
  - `phase`: current stage of the guided flow
  - `acceptedClarifications`: recorded answers that changed planning content
  - `outstandingQuestions`: unresolved high-impact ambiguities
  - `nextRecommendation`: recommended follow-up step for the user
- **Validation rules**:
  - Must reference one active feature workspace at a time
  - Must not advance phases while critical clarifications remain unresolved
  - Must maintain traceability from accepted clarification to updated artifact
- **State transitions**:
  - `started` -> `clarifying` -> `planning` -> `ready-for-tasks` -> `completed`
  - `started` or `planning` -> `interrupted` -> `resumed`

## PlanningArtifact

- **Purpose**: Represents a markdown artifact generated by the workflow.
- **Key fields**:
  - `artifactType`: `spec`, `plan`, `research`, `data-model`, `quickstart`, `contract`, or `tasks`
  - `ownerWorkspace`: feature workspace that owns the artifact
  - `path`: repository-relative markdown path
  - `status`: draft, current, superseded, or missing
  - `sourceInputs`: upstream inputs used to generate or update the artifact
  - `lastUpdatedPhase`: workflow phase that most recently changed the artifact
- **Validation rules**:
  - Must be stored as markdown
  - Must belong to exactly one feature workspace
  - Must preserve traceability to its upstream source inputs

## Relationships

- One **RepositorySetup** manages many **FeatureWorkspace** records.
- One **FeatureWorkspace** owns many **PlanningArtifact** records.
- One **FeatureWorkspace** can have many **SDDSession** records over time, but only one active session at a time.
- **SDDSession** updates **PlanningArtifact** records and advances the **FeatureWorkspace** status.
